# План пополнения склада филиала — Гардеробные системы

Это статическая онлайн-версия расчёта для филиалов. Пользователь загружает только два актуальных Excel-отчёта:

1. **Анализ материальных запасов** — для свободного остатка с подтверждением и товара в пути.
2. **Планы на отгрузку с завода** — для уже созданных поставок в выбранный филиал.

Исторический спрос, справочник 177 SKU гардеробной системы, статусы и кратность упаковки встроены в `model-data.js`. Текущая версия спроса обновлена по отгрузкам **2025** и **2026 по 08.07.2026**: **35 филиалов**, 5 177 строк спроса по связке филиал × SKU. Филиал «Узловая» исключён из пользовательского выбора.


## Логика расчёта среднемесячного спроса

Среднемесячный спрос считается **отдельно для каждого филиала и каждого SKU**, чтобы не смешивать разные профили продаж филиалов.

Используются только полные месяцы:

- 2025 год: январь–декабрь, 12 месяцев;
- 2026 год: январь–июнь, 6 месяцев;
- частичный июль 2026 из файла «по 08.07.2026» исключён, потому что неполный месяц занизил бы среднемесячный спрос.

Формула:

```
Ср.мес. спрос = Ср.мес. 2025 × 41,18% + Ср.мес. 2026 × 58,82%
```

Вес 2026 года выше, потому что это более свежий спрос. Пропорция получена из прежней логики модели: веса 2025 = 0,35 и 2026 = 0,50 нормализованы без 2024 года.

Пример: если филиал СПБ по артикулу продавал 76,25 шт./мес. в 2025 и 59,5 шт./мес. в 2026, то обновлённый спрос:

```
76,25 × 41,18% + 59,5 × 58,82% = 66,40 шт./мес.
```

## Как работает расчёт

1. Логист выбирает филиал, дату отгрузки с завода, экспертное число дней в пути и целевой срок хранения МЗ в днях.
2. Из двух отчётов автоматически отбираются только товары, присутствующие в справочнике ПРАКТИК Home. Приоритет сопоставления — `ID товара`; для плана предусмотрен резервный поиск по артикулу.
3. Для каждого SKU рассчитываются:

   - **Целевой МЗ, шт.** = среднемесячный спрос / 30,44 × целевой срок хранения, округление вверх до целых штук.
   - **Прогноз до прихода** = среднемесячный спрос / 30,44 × дни до расчётной даты прихода.
   - **Расчётная потребность** = целевой МЗ + прогноз до прихода − свободно с подтверждением − в пути − ранние планы.
   - **К заказу** = положительная расчётная потребность, округлённая вверх до кратности упаковки.

4. «Планы до выбранной даты» — отгрузки с завода **до** выбранной даты новой поставки; они уменьшают потребность, поскольку должны быть выполнены раньше. «Планы с выбранной даты и позже» выводятся для контроля, но не уменьшают текущую заявку.

Расчёт полностью ведётся в штуках: цены, скидки, РЦ, объём МЗ в м³ и план продаж не используются.

## Контроль входных данных

В интерфейсе добавлен блок предупреждений. Он появляется, если есть риск, что расчёт построен на неправильных вводных:

- в плане отгрузок нет строк по выбранному филиалу, но есть строки других филиалов;
- в отчёте МЗ найдено отдельное поле филиала и филиал не совпадает с выбранным;
- в отчёте МЗ не найдено или найдено слишком мало SKU гардеробной системы;
- есть строки плана с некорректной датой;
- дата отгрузки или дата прихода нарушают календарную логику;
- целевой срок хранения задан слишком низким или слишком высоким.

Критический случай — отсутствие SKU гардеробной системы в отчёте МЗ. В этом случае расчёт останавливается, потому что остатки будут некорректными. Остальные предупреждения не блокируют расчёт, но требуют проверки логистом.

## Требования к входным файлам

### Анализ МЗ

Нужные поля в заголовке:

- `ID товара`;
- `Своб_подт` или «Свободно с подтверждением»;
- `Sum-В пути` или «В пути».

Этот отчёт должен быть сформирован **по тому же филиалу**, который выбран в форме.

### Планы отгрузок

Нужные поля:

- `Фил` или `Филиал`;
- `Дата плановая`;
- `Кол` / `Кол-во` / `Количество`;
- `ID товара` или `Артикул`.

В файле могут быть строки любых товарных направлений и разных филиалов. Инструмент автоматически оставляет только гардеробные системы выбранного филиала.

## Запуск

### Для проверки на компьютере

Откройте `index.html` в браузере. Для чтения и выгрузки XLSX страница по умолчанию подключает браузерную библиотеку XLSX через CDN; при закрытом внешнем доступе потребуется разместить эту библиотеку на внутреннем сервере и заменить строку подключения в `index.html` на локальный путь.

### Для размещения во внутренней сети

Загрузите содержимое этой папки на внутренний веб-сервер или в корпоративный статический хостинг. Серверу не нужны база данных и серверная логика: отчёты обрабатываются локально в браузере и на сайт не передаются.

Минимальная структура:
```
/gs-replenishment-online/
  index.html
  app.js
  model-data.js
  admin/update_model.py
```

## Обновление истории и справочника

Филиалам менять файл истории не нужно. Администратор может обновить справочник и спрос двумя способами.

### Вариант 1. Из рабочей Excel-модели


```bash
python admin/update_model.py "Новая модель пополнения.xlsx" "model-data.js"
```

Скрипт ожидает листы `Лист1` и `Спрос филиалов` с той же логикой колонок, что в текущей модели. После замены `model-data.js` достаточно обновить файлы на сервере.

### Вариант 2. Обновить спрос и сезонность напрямую из выгрузок отгрузок

В текущей версии спрос и сезонные коэффициенты обновлены именно этим способом: из файлов `отгрузки 2025 по ГС.xlsx` и `отгрузка 2026 год по 08.07.2026.xlsx`.

Рекомендуемый скрипт для следующего обновления:

```bash
python admin/update_demand_and_seasonality_from_shipments.py "отгрузки 2025 по ГС.xlsx" "отгрузка 2026 год по 08.07.2026.xlsx" "model-data.js"
```

Отчёт о предыдущей операции сохранён в `admin/demand_update_report_2026-07-08.json`, а список артикулов из отгрузок, которых нет в справочнике пополнения, — в `admin/excluded_articles_not_in_product_master_2026-07-08.csv`.

## Что изменилось в v8

- Добавлен метод расчёта спроса **«Сезонный спрос»**. Он установлен по умолчанию.
- В параметрах поставки появился переключатель метода:
  - **Сезонный спрос** — рекомендуемый режим;
  - **Средний спрос** — прежняя логика без сезонности.
- Сезонная логика работает не только по месяцу отгрузки, а по реальному периоду покрытия:
  - прогноз до прихода считается по дням от текущей даты до даты прихода;
  - целевой МЗ считается по дням от даты прихода на выбранный срок хранения МЗ.
- Для каждого дня применяется коэффициент того месяца, в который этот день попадает. Если период хранения пересекает два месяца, программа учитывает оба месяца пропорционально дням.
- Базовый спрос по-прежнему считается отдельно по каждому филиалу и SKU. Сезонность накладывается сверху на этот базовый спрос.
- Для крупных филиалов используется сезонность конкретного филиала. Для филиалов с меньшим объёмом статистики применяется смешанная сезонность: часть филиальной сезонности и часть общей сезонности компании.
- Коэффициенты сезонности рассчитаны по отгрузкам 2025 года и полным месяцам 2026 года. Частичный июль 2026 исключён из расчёта сезонности.
- В таблицу добавлен столбец **«Коэф. сезона»**, а в KPI добавлена карточка **«Сезонность месяца прихода»**.
- В выгрузку XLSX/CSV добавлен коэффициент сезонности и источник сезонности.

### Формула сезонного расчёта

Базовый среднемесячный спрос:

```text
Базовый спрос = спрос конкретного SKU в конкретном филиале, шт./мес.
```

Сезонный дневной спрос:

```text
Дневной спрос = Базовый спрос × коэффициент сезона месяца / количество дней в месяце
```

Целевой МЗ:

```text
Целевой МЗ = сумма сезонного дневного спроса с даты прихода на целевой срок хранения МЗ
```

Расчётная потребность:

```text
Расчётная потребность = Целевой МЗ + прогноз до прихода − свободно с подтверждением − в пути − планы до выбранной даты
```

## v11 — исправление даты по умолчанию

Исправлено заполнение поля «Плановая дата отгрузки с завода» при открытии страницы.

Причина: раньше дата формировалась через `toISOString()`, из-за чего в часовых поясах восточнее UTC браузер мог показывать предыдущий календарный день. Теперь дата формируется по локальной дате браузера пользователя: год, месяц и день берутся напрямую из `new Date()` без UTC-конвертации.

Расчётная логика, сезонность, проверки и выгрузка XLSX/CSV не менялись.
